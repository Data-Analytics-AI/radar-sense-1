# SnapFort - Real-Time Fraud & AML Intelligence Platform

## Overview
SnapFort is an enterprise fraud detection and AML (Anti-Money Laundering) transaction monitoring dashboard for financial services institutions. It provides real-time transaction monitoring, alert management, case investigation, and AI-assisted fraud analysis. The platform aims to enhance financial crime prevention capabilities for financial services institutions.

## System Architecture
The application features a React 18 frontend with TypeScript, Vite, Tailwind CSS, and Shadcn UI. The backend is an Express.js API server, primarily for proxying AI chat requests. Routing is handled by react-router-dom v6, and state management utilizes TanStack React Query v5. The system currently uses client-side mock data.

Key features include:
- **Dashboard (Fraud Command Center)**: A 10-section command center with system status header, 6 expandable KPI cards, live threat monitor, intelligent trend analytics (channel-segmented volume, fraud trend with anomaly markers, risk pyramid), AI fraud insights panel, investigator operations (enhanced live feed with device icons/geo flags/risk tooltips, priority-sorted alerts with SLA timers, risky entities), model health snapshot, analyst performance, fraud network mini-preview, and compliance status. Data from `src/lib/dashboard-data.ts`.
- **Real-time Monitoring**: Live transaction feed.
- **Alert & Case Management**: Tools for managing alerts and tracking case investigations, including a comprehensive workspace for individual cases with timelines, entities, evidence, and notes.
- **Transaction Analysis**: Search and detailed analysis of transactions.
- **Analytics & Reporting**: Charts and reporting features, including an enhanced executive summary with clickable insights and actionable recommendations, and detailed metrics across fraud, AML, models, rules, geography, channels, users, operations, and audit.
- **Graph Network Analysis**: An investigation-grade workspace for analyzing fraud rings with interactive SVG graphs, pathfinding, and node detail drawers.
- **AI Assistant**: A context-aware AI investigation assistant with structured response rendering supporting six intent-specific views (transaction analysis, customer analysis, SAR drafting, analytics summary, case summary, general answers). It uses a hybrid intent detection system and a markdown-to-structured parser.
- **AI Layer (Azure OpenAI)**: Server endpoint `POST /api/ai/analyze` (kind-routed: `kyc_summary | fraud_explain | str_draft | risk_reasoning`) with Azure OpenAI primary + OpenAI fallback, request validation (kind allowlist, 64KB context cap, 256KB body cap). Client helper `src/lib/ai.ts`. UI integration: "Summarize KYC (AI)" in Customer Onboarding drawer, "Explain with AI" in Alert Details Panel, "AI-Draft Narrative" in Regulatory Reporting (pre-fills STR narrative), "Explain Risk (AI)" in Case Investigation Risk Score tab.
- **Compliance Alerts**: Alert type union extended with `kyc_risk | pep_match | sanction | edd_required`. `getComplianceAlerts()` in compliance-data.ts derives ~18 alerts from existing customers/screening matches/EDD cases. Alerts page merges them with fraud/aml alerts, has color-coded type badges (orange/purple/red/amber) and a "Compliance" quick-filter pill. Alert detail panel renders a "Compliance Context" section for the 4 new types.
- **Case Investigation Tabs**: 7 new tabs added at the top: KYC Profile (masked BVN/NIN reveal), Risk Score (gauge + breakdown table via `computeRiskScore`), Transactions (top 20 for customer), Network (inline SVG mini-graph), Documents (verified status), EDD Notes (from `getEddCases`), Reg Flags (STRs/CTRs/fraud entries/screening matches). Existing Timeline/Entities/Evidence/Notes/Export tabs preserved.
- **Rules Engine**: A comprehensive visual builder for fraud detection rules with condition trees, simulation, and version history.
- **Model Management**: Enterprise ML model lifecycle management with 5 sections: Production Model Control Bar (KPIs, promote/rollback/retrain actions), Performance & Risk Impact (detection trends, loss analysis, PR trade-off, confusion matrix), Drift/Data Quality Monitoring (feature drift PSI, data quality checks, "What Changed" insights), Model Registry (searchable/filterable cards with 5-tab detail drawer: Summary, Explainability, Operations, Governance, Evaluation), and Safe Actions (retrain wizard, compare mode, confirmation dialogs). Includes RBAC UI gating for admin/ML engineer/risk analyst/auditor roles. Components: `src/pages/Models.tsx`, `src/lib/models-data.ts`, `src/components/models/`.
- **User & Access Management**: Comprehensive user management with roles, permissions, audit logging, and lifecycle management. **Persisted to Huawei RDS Postgres** via `users`, `roles`, `user_role_assignments`, and `audit_log` tables. API: `/api/users` (GET/POST/PATCH), `/api/roles` (GET/POST/PATCH/DELETE), `/api/role-assignments` (GET), `/api/users/:userId/roles` (POST + DELETE :roleId), `/api/audit-log` (GET filters: actor/target/targetType/since/until/limit, POST). User mutations and customer create/action mutations append audit entries (actor headers `x-actor-user-id` / `x-actor-name`). New "Audit Log" tab on Users page provides filterable viewer (actor, target type, search, date).

  Server enforcement: new `requirePermission(key)` Express middleware resolves the actor (`x-actor-user-id` header) → roles → union of permissionKeys, then 401/403s when missing. Applied to all mutating routes — users/roles/role-assignments (manage_users / manage_roles), rules incl. `/:id/versions` and `/:id/simulate` (create_rules / edit_rules / delete_rules / simulate_rules), alerts POST/PATCH and `/:id/notes` (manage_alerts), cases POST/PATCH and `/:id/notes|timeline|entities|evidence` (create_cases / manage_cases), regulatory-reports POST/PATCH/`transactions`/`artifacts` (file_reports), fraud-register POST/PATCH/`timeline` (log_fraud_incident), and **models** lifecycle (`/promote`, `/rollback`, `/threshold` → `deploy_models`; `/retrain` → `retrain_models`) — the legacy `checkModelRole` gate has been removed in favour of the shared permission map. Admin role is treated as a superuser.

  Client surface: `GET /api/users/me/permissions` returns `{userId,name,roles,permissions}` for the active actor; `GET /api/users/demo-accounts` lists the seven canonical accounts. New `usePermissions()` hook (`src/hooks/usePermissions.ts`, explicit fetch URL + actor-keyed cache) exposes `can(key)`, `hasRole(role)`, current name and roles. The header (`src/components/layout/Header.tsx`) replaces the static "John Doe" avatar with an "Act as (demo)" switcher dropdown that lists every demo account, calls `setCurrentUser()`, and invalidates every TanStack query so subsequent fetches use the new actor header. Buttons gated client-side: Users page "Add User" + per-user Suspend/Deactivate/Activate/Reactivate/Resend-Invite quick-actions and Reset-MFA / Force-Password-Reset / Lock / Unlock + MFA toggle (manage_users), "Manage Roles" header button + per-user role checkboxes (manage_roles), Rules page Create-Rule buttons (create_rules), Save-Rule (edit_rules / create_rules), Run-Simulation (simulate_rules), Alerts panel Assign/Escalate/Add-Note (manage_alerts) and Create-Case-from-Alert + Cases page "New Case" (create_cases), "Submit / Submit to NFIU" in Regulatory Reporting (file_reports), Models page Promote/Rollback/Retrain (deploy_models / retrain_models — replaces the hardcoded `CURRENT_USER_ROLE = 'admin'`), Settings Save/Reset on every section (manage_settings), and Case Investigation status-change buttons + Post Note / Link Entity / Upload Evidence / Save EDD Note (manage_cases). Server `resolveActorPermissions()` derives effective roles from the `user_role_assignments` table (with a legacy `users.roles[]` fallback) so assignment mutations cannot drift from runtime permissions. Out of scope: real authentication and role-edit UI (#22), full signed-in-user capture (#20).

The frontend and backend run concurrently, with Vite proxying `/api` requests to the Express server.

- **Transactions (Live Database)**: ~2000 historical transactions seeded into the `transactions` table at boot (server/seed.ts), keyed to real customer IDs (IND-/BIZ-). Server endpoints: `GET /api/transactions` (filters: customerId, search, type, status, riskLevel, riskMin, riskMax, since, limit), `GET /api/transactions/:id`, `POST /api/transactions`. React Query hook `src/hooks/use-transactions-api.ts` (useTransactionsQuery / useTransactionQuery / useCreateTransaction). Wired consumers: `src/pages/Transactions.tsx`, `src/pages/LiveMonitoring.tsx` (polls every 4s while live), `src/components/dashboard/LiveTransactionFeed.tsx` (polls every 5s), and Case Investigation's "Top 20 transactions" tab now query Postgres. Customer Profile's synthetic graph data still uses `getSyntheticTransactions()` (out of scope for this migration).

- **Case Evidence in OBS**: Case Investigation Evidence tab uploads files to Huawei OBS via the presigned-PUT flow. Each evidence row on `cases.evidence` carries a `storageKey` under the `case-evidence/` namespace; the server `POST /api/cases/:id/evidence` handler refuses orphaned metadata (zod-required `storageKey`, namespace check, `headObject` round-trip), trusts the bucket for `fileSize`/`mimeType`, and appends a server-stamped "uploaded" custody-chain entry referencing the OBS key + byte count. Evidence cards expose an "Open / Download" button that mints a 5-minute signed GET URL via `/api/uploads/sign-download`. Legacy seed rows without `storageKey` render as "metadata only — no stored file" and hide the open action. Client helpers in `src/lib/obs-upload.ts` (`uploadFileToObs`, `getSignedDownloadUrl`); shared namespace union in `src/types/obs.ts`.

- **Huawei OBS (Object Storage)**: Live connection backing KYC documents, case evidence, AI chats, and generated reports. Backend abstraction `IObjectStore` in `server/object-store.ts` with two implementations: `HuaweiObjectStore` (production, wraps `esdk-obs-nodejs`) and `MemoryObjectStore` (dev/CI fallback when `HUAWEI_OBS_*` secrets are missing — logs a loud WARN at boot, ERROR if `NODE_ENV=production`). Selected at boot via `getObjectStore()`. Surface: `putObject`, `getObject`, `deleteObject`, `headObject`, `getSignedUploadUrl`, `getSignedDownloadUrl`. Endpoints: `GET /api/system/obs-health` performs a real round-trip (put a small probe → sign GET → fetch the URL → delete), returns `{ok, backend, bucket, endpoint, region, latencyMs, steps:{put,sign,fetch,delete}}`; `POST /api/uploads/sign` issues a 5-minute presigned PUT URL with namespace allow-list (`customer-documents | case-evidence | reports | ai-chats`; `system-health` reserved for probes), 25MB max size, deterministic key `{namespace}/{yyyy}/{mm}/{uuid}-{safeFileName}`; `POST /api/uploads/sign-download` mints short-lived GET URLs (defence-in-depth namespace check). All SDK errors sanitized to user-safe messages with full details logged server-side; AK/SK never echoed and only flow into the SDK client. Settings → Database & Storage shows the "Huawei OBS — Object Storage" card with backend, round-trip latency, per-step timings, bucket/endpoint/region, and a "Test Connection" button.

## Deployment
The project ships as a single Docker image. See [`DEPLOYMENT.md`](./DEPLOYMENT.md) for the full Huawei Cloud ECS + SWR walkthrough (build → push to SWR → pull/run on ECS, env-var table, optional Nginx + Let's Encrypt, optional systemd unit, rollback). Runtime contract: container exposes port 5000, serves the React app + Express API in one process, connects out to Huawei RDS (Postgres) and Huawei OBS, and exposes `GET /api/health` for liveness checks. `Dockerfile` is multi-stage (Node 20 slim builder → slim runtime), `.dockerignore` keeps the image small, and `.env.example` lists every required variable.
