You are a Senior Full-Stack Engineer improving an existing SnapFort (Fraud Detection) web app.

GOAL
Optimize the following pages:
1) Transactions page
2) Live Monitoring page
3) Alerts page

Make every transaction/alert row clickable, and when the chevron/arrow at the far right of a row is clicked, a full “Details” view pops up (drawer or modal). This must work consistently across all three pages.

IMPORTANT UX BEHAVIOR (MUST IMPLEMENT)
A. Row click:
- Clicking anywhere on a row (except interactive controls like checkboxes, menus, filters) should open the details view for that item.
- The row should show hover + focus states (keyboard accessible).

B. Chevron click (arrow beside row):
- Clicking the chevron should expand/open the details view without navigating away.
- The chevron should rotate to indicate expanded state.
- This should NOT trigger row-click twice (stopPropagation / prevent double-open).

C. Keyboard accessibility:
- Row is focusable (tabIndex=0) and opens on Enter/Space.
- Chevron is a real button element with aria-expanded, aria-controls.

D. Detail view:
- Use a right-side drawer if screen width >= 1024px.
- Use a modal bottom sheet (or full-screen modal) on mobile.
- Show a skeleton loader while details are fetched.
- Include a close button and ESC to close.
- Opening details should update URL (optional but preferred) via query param:
  - /transactions?selected=TXN-123
  - /alerts?selected=ALT-123
  - /live-monitoring?selected=TXN-123
- On refresh, the selected item auto-opens if the query param exists.

E. Data fetching strategy:
- List endpoints remain as-is.
- Add “details” endpoints (if not existing) for each entity type and fetch by ID:
  - GET /api/transactions/:id
  - GET /api/alerts/:id
  - (Live Monitoring uses transactions stream; details should call /api/transactions/:id)
- Use caching so opening the same item again does not refetch unless stale.
- Handle errors with a friendly inline message + retry.

SCOPE (PAGES)
1) Transactions page:
- Table rows are currently rendered with a rightmost arrow icon.
- Implement clickable row + chevron behavior.
- Details drawer shows:
  - Transaction summary (ID, timestamp, channel, type, amount, currency, status)
  - Customer (ID, name if available, risk tier)
  - Merchant (name, MCC)
  - Risk score + confidence

  - Rules hit (if available)
  - Geo/device/IP info if available
  - Raw payload (collapsible JSON viewer)
  - “Create Alert” or “Open Case” CTA buttons (if present in system)

2) Live Monitoring page:
- Live stream list/table should behave like Transactions page.
- New rows arriving should not break selection state.
- If an item is currently selected, keep drawer open; highlight the selected row.

3) Alerts page:
- Alerts list rows are clickable + arrow opens details.
- Details drawer shows:
  - Alert summary (ID, type Fraud/Rule/Model, severity, status, created time)
  - Linked transaction(s) and customer
  - Contributing factors / risk drivers
  - Investigation timeline (status changes, assignments, notes if available)
  - Actions: Assign, Change Status, Create Case, Add Note (wire to existing endpoints or create stub endpoints if missing)

UI COMPONENTS (IMPLEMENT CONSISTENTLY)
